# 10 · Conventions

Code style, file organization, and rules that keep the codebase readable as it grows. Short on rules, opinionated where it matters.

---

## File and folder layout

```
src/
├── app/                       Next.js App Router routes only
├── modules/                   feature modules — see 02-architecture.md
├── lib/                       cross-cutting infrastructure
│   ├── spotify/
│   ├── claude/
│   ├── db/
│   ├── theme/
│   ├── ui/                    primitive UI components
│   └── utils/
├── styles/                    global stylesheet, Tailwind directives
└── types/                     shared types not owned by any module
```

**Rules:**

- One feature → one folder under `modules/`. Each folder has at minimum `index.ts`, `repository.ts`, and `ui/`.
- Generic helpers go in `lib/utils/` only when used by 2+ modules. Single-use helpers stay co-located.
- No `src/components/` global folder. Primitives live in `lib/ui/`; module-specific components live in their module.
- Route handlers (`app/api/.../route.ts`) are thin orchestrators. No business logic in routes.

---

## Imports

- Path alias: `@/` → `src/`.
- Modules expose their public API through their `index.ts`. **Always import from the index, never from internals.**
  - Good: `import { getTodayPick } from "@/modules/morning-pick";`
  - Bad: `import { computeScore } from "@/modules/morning-pick/algorithm";`
- Modules **do not import from other modules.** Cross-module needs go through the database, through `lib/`, or through the Spin Companion tool dispatcher.
- Group imports: external packages, then `@/lib/...`, then `@/modules/...`, then relative. Blank line between groups.

---

## TypeScript

- `strict: true` in `tsconfig.json`. Always.
- No `any`. If you really need a wide type, use `unknown` and narrow it.
- Prefer `type` aliases over `interface` unless you need declaration merging (almost never).
- Function signatures get explicit return types when the function is exported. Inferred returns are fine for internal helpers.
- Use discriminated unions for variant data (e.g., `TriageAction` in capture-inbox). Don't reach for class hierarchies.
- Database row types are imported from `@/lib/db/types` (generated by Supabase CLI). Never redeclare them.

---

## React

- Server Components by default. `"use client"` only when needed.
- Suspense boundaries around any data-fetching island so the rest of the page doesn't block.
- No `useEffect` for data fetching. If a component needs server data, fetch it in a Server Component or use a server action.
- Co-locate component-specific hooks in the same file (or right next to it). No global `hooks/` folder.
- Component names: `PascalCase`. File names match the default export. One component per file unless they're tightly bound.
- Props: prefer destructuring at the top of the component over inline access. Spread props only on intentional pass-through wrappers.

---

## Styling

- Tailwind utilities only. No `style={{...}}` except when binding to ambient CSS variables (and even then, prefer extending Tailwind config).
- Custom semantic tokens in `tailwind.config.ts` reference `var(--ambient-*)`. Components use the tokens, not raw colors:
  - Good: `bg-ambient text-ambient-fg border-ambient-edge`
  - Bad: `bg-amber-50 text-stone-900`
- Class names ordered by intent: layout → spacing → typography → color → state. The `prettier-plugin-tailwindcss` plugin enforces this.
- No `cn()` utility wrapper unless you actually need conditional classes. `clsx` directly is fine.

---

## Async and errors

- All async functions return either a value or throw. No "ok/err" tuples.
- Errors at module boundaries are typed:
  ```ts
  export class SpotifyAuthError extends Error { ... }
  ```
- API routes catch known error types and return appropriate status codes; unknown errors return 500 and log.
- Never swallow errors. If you intentionally don't act on an error, comment why.

---

## Database access

- Modules access the DB through their own `repository.ts`. Repositories use the typed Supabase client from `@/lib/db`.
- Repository functions accept primitives (ids, options) and return typed objects. They never return raw Supabase response objects.
- All queries go through RLS — we never use the service role key in user-facing code paths. The service role key is only used in cron jobs and admin scripts.
- Migrations are SQL files in `supabase/migrations/` named `YYYYMMDDHHMMSS_descriptive_name.sql`. Run via the Supabase CLI.

---

## Spotify and Claude clients

- Always go through `lib/spotify/client.ts` and `lib/claude/client.ts`. No direct calls to either provider's SDK from modules.
- These clients handle: auth, rate limiting, retries, telemetry. Bypassing them means bypassing all four.

## Naming

- Functions: `verbNoun`. `getTodayPick`, not `todayPick` or `pick`. `recompute`, not `recalc`.
- Booleans: `isFoo`, `hasFoo`, `canFoo`. No `flag` or `enabled`.
- Time fields: explicit. `playedAt` (timestamp), `daysSinceLastPlay` (number). Never `lastPlay` (ambiguous unit).
- Database columns: `snake_case`. TypeScript fields: `camelCase`. The repository layer translates.
- Constants: `SCREAMING_SNAKE_CASE`. Co-located with use unless shared.

---

## Comments

- Comments explain **why**, not what. The code shows what.
- A function doing something non-obvious deserves a one-paragraph header comment. Most don't need any.
- TODO comments include initials and a date: `// TODO(et 2026-05-04): handle re-resolution after manual edit`.
- Delete dead code. Never comment it out "in case."

---

## Tests

- `vitest` for unit and module tests. Files live next to the code: `algorithm.ts` → `algorithm.test.ts`.
- Integration tests for repositories use a dedicated test database (Supabase local dev or a separate cloud project).
- E2E tests (Phase 2+) live in `e2e/` and use Playwright.
- Test the boundaries (public API of a module), not the internals.
- A unit test that mocks more than two things is probably testing the wrong layer.

---

## Git

- Branch names: `phase-N-short-description` for roadmap work; `fix/short-description` for bugfixes; `docs/short-description` for doc-only changes.
- Commit messages: short imperative subject under 72 chars, optional body. No emoji prefixes, no Conventional Commits ceremony.
- One feature per PR. PR title mirrors the commit; description links the relevant module spec.
- Squash on merge.

---

## Secrets

- Never commit secrets. `.env.local` is gitignored; `.env.example` lives in the repo with empty values and short comments.
- Required env vars are documented in `docs/03-tech-stack.md` and validated on app boot via a `lib/env.ts` zod schema.

---

## Accessibility (recap from design system)

- All interactive elements have visible focus states using `--ambient-accent`.
- All images have `alt` attributes. Album art is `alt=""` (decorative); other images describe themselves.
- Motion respects `prefers-reduced-motion`.
- Keyboard navigation is the default; mouse-only interactions are forbidden.

---

## What "minimalism" means in code

The same way it does in the UI. Each module does one thing. Each function has one job. Each test verifies one behavior. If a file is doing two things, it should be two files.

When you reach for cleverness, ask whether plain code would do. It almost always does.
